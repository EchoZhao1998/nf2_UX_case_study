
For a 4-person student group, a comprehensive app case study typically takes 4 to 6 weeks to complete from start to finish. This translates to roughly 40 to 60 total hours of work per person, spread out to accommodate classes and schedules.
Dividing the work efficiently among four people speeds up production, but requires strict coordination. [1, 2] 
## Recommended 4-Week Timeline & Work Distribution## Week 1: Research & Discovery (10–12 hours)

* Goal: Define the problem, analyze competitors, and understand users.
* Person A & B: Conduct 3–5 user interviews and synthesize data into user personas.
* Person C & D: Perform competitive analysis and map out the initial user journey. [3, 4, 5] 

## Week 2: Information Architecture & Wireframing (12–15 hours)

* Goal: Map out the app structure and create low-fidelity layouts.
* Person A: Create the final user flow and app sitemap.
* Person B & C: Design low-fidelity wireframes in Figma for all core app screens.
* Person D: Run quick usability tests on wireframes and document the feedback. [6, 7] 

## Week 3: UI Design & Prototyping (15–20 hours)

* Goal: Build the visual identity and turn wireframes into an interactive app.
* Person A & B: Build the component library (buttons, typography, colors) and design high-fidelity screens.
* Person C: Turn the high-fidelity screens into a working prototype with Figma transitions.
* Person D: Conduct final prototype testing with users and note required fixes. [8, 9, 10, 11] 

## Week 4: Case Study Assembly & Presentation (10–12 hours)

* Goal: Tell the story of your project in a clean, scannable Figma presentation file.
* Person A & B: Write the case study narrative (Problem, Solution, Lessons learned).
* Person C & D: Clean up the Figma canvas, create device mockups, and polish the final presentation deck. [12] 

## Top Tips for 4-Person Figma Groups

* Use a Design System: Agree on a shared color palette and typography scale on day one to avoid messy, mismatched screens.
* Use Components: Build reusable headers, buttons, and navigation bars so changes update globally for everyone.
* Section the Canvas: Divide your Figma file into clear pages (e.g., 01_Research, 02_Wireframes, 03_Final_UI) so team members do not accidentally edit each other's work. [13, 14]

For a student project tackling Neurofibromatosis Type 2 (NF2), the case study must center on high-stakes data organization and accessible UI design [1]. Because NF2 causes benign tumors on the auditory and cranial nerves, patients experience progressive hearing loss, vision impairment, and balance issues [1]. Your group's app must treat accessibility (WCAG compliance) as a core product feature, not an afterthought.
Here is a specialized 4-to-6-week roadmap tailored specifically for this medical folder app.
------------------------------
## Step-by-Step Project Roadmap## Week 1: Research, Empathy & Clinical Context

* Goal: Understand NF2 patient limitations and map out medical document types (e.g., MRI scans, audiograms, genetic reports, surgery summaries) [1].
* Person A & B: Research NF2 symptom progression [1]. Define 2 user personas: a newly diagnosed patient (overwhelmed, tracking baseline data) and a long-term patient (managing progressive hearing/vision loss) [1].
* Person C & D: Audit existing patient portals (like Epic MyChart). Document where they fail complex chronic patients (e.g., rigid categorization, poor mobile document scanning).

## Week 2: Medical Architecture & Accessibility Planning

* Goal: Build a logical medical folder structure and establish strict accessibility guidelines.
* Person A: Create a secure, intuitive Information Architecture (IA). Map how a user uploads, tags (by body part/date), and shares a report with a specialist.
* Person B & C: Design low-fidelity wireframes in Figma. Focus heavily on massive tap targets, clear dashboard layouts, and a "Timeline" view of tumor growth reports [1].
* Person D: Establish the design system's accessibility guardrails (minimum 16pt body text, WCAG AAA color contrast ratios, screen-reader optimized layouts).

## Week 3: High-Fidelity UI & Figma Prototyping

* Goal: Turn wireframes into a clean, calming, and highly functional medical application.
* Person A & B: Build the high-fidelity UI using a calming medical color palette (blues/grays). Ensure all interactive elements have strong visual indicators (underlines, high contrast borders).
* Person C: Build the interactive Figma prototype. Focus on three critical user flows:
1. Fast-uploading a multi-page PDF/photo of an MRI report.
   2. Filtering reports by "Audiograms" or "Brain/Spine MRIs".
   3. Generating a quick "Emergency Health Summary" QR code for ER doctors.
* Person D: Simulate user testing. Run the prototype through visual impairment filters (such as blurry vision or tunnel vision plugins in Figma) to verify readability.

## Week 4: Case Study Storytelling & Presentation

* Goal: Package the project into a compelling Figma case study presentation deck.
* Person A & B: Write the core narrative. Emphasize the Why—how an organized folder reduces clinical anxiety for a patient managing an incurable disease [1].
* Person C & D: Design the final presentation slides in Figma. Use realistic, mock medical data. Show side-by-side evolution of your UI from wireframe to final accessible layout.

## Figma Team Best Practices for this Project

* Enforce Auto Layout: Use Auto Layout on every single button and list item. If you need to scale text sizes up for accessibility testing, the UI will automatically expand without breaking.
* Real Data Only: Do not use "Lorem Ipsum." Use realistic dummy medical text (e.g., "MRI of the internal auditory canals shows a 1.2cm vestibular schwannoma..."). It anchors the case study in true clinical empathy [1].
